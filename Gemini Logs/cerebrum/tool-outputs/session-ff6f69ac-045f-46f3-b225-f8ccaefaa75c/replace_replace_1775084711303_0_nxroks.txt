Introduction\n\n### 1.1 The Gap Between Knowledge Graphs and Language Models\n\nKnowledge Graphs and Large Language Models represent two fundamentally\ndifferent approaches to knowledge representation and reasoning.\n\nKnowledge Graphs store knowledge explicitly: entities as nodes, relationships\nas typed edges, facts as (subject, predicate, object) triples. This makes\nthem precise, verifiable, and updatable without retraining. However, they\ncannot reason beyond what is explicitly stored. Multi-hop inference — \"Marie\nCurie discovered Polonium, Polonium is radioactive, therefore Marie Curie\ndiscovered a radioactive element\" — requires either hardcoded graph traversal\nqueries or external reasoning systems.\n\nLarge Language Models store knowledge implicitly in billions of weight\nparameters. They can reason, generalize, and synthesize across domains. But\nthis implicit representation is opaque, cannot be updated without expensive\nfine-tuning, and is prone to hallucination — generating plausible-sounding\nbut incorrect facts with no mechanism for ground-truth verification.\n\nThe field has responded with hybrid approaches: Retrieval-Augmented Generation\n(RAG), Knowledge-Graph-augmented LLMs, and GraphRAG. In all of these, the KG\nis a retrieval store and the LLM does the reasoning. The KG remains passive.\n\n**CEREBRUM inverts this relationship.** The KG reasons. The LLM, if present,\nonly generates natural language from the KG's output. Every inference step is\na graph traversal, every conclusion is a path, and every path can be verified.\n\n### 1.2 The Core Observation\n\nA Transformer's power comes from three mechanisms working together:\n\n1. **Multi-head attention**: different heads specialize on different\n   relational aspects of the input (syntactic, semantic, long-range, etc.)\n2. **Deep composition**: each layer builds on the previous, allowing complex\n   multi-step reasoning\n3. **Positional awareness**: the model knows where each token sits relative\n   to others\n\nWe observe that Knowledge Graphs have natural analogs for all three:\n\n1. **Community structure** serves the role of attention heads: nodes within\n   a community have strong mutual relevance; communities specialize on\n   different conceptual domains.\n2. **BFS hop depth** serves the role of layer depth: each hop is one step of\n   composed reasoning.\n3. **Graph-structural features** (PageRank, betweenness, degree) serve the\n   role of positional encoding: they tell the model where each entity sits in\n   the global information landscape.\n\nThe question is: can these analogs be made operational — not merely\nmetaphorical? We argue yes, and demonstrate the architecture to do so.\n\n### 1.3 Contributions\n\nThis paper makes three primary contributions:\n\n1. **The CEREBRUM architecture**: a complete mapping of Transformer components\n   to KG operations, enabling multi-hop reasoning via graph traversal alone.\n\n2. **Community-Structured Attention (CSA)**: a novel attention mechanism using\n   community membership as a soft global constraint on graph traversal,\n   bridging the gap between local GAT-style attention and global Transformer-\n   style attention.\n\n3. **Dual-Signal Community Fusion (DSCF)**: a novel community detection\n   algorithm combining LPA majority-vote and modularity gain simultaneously\n   at each node update, producing communities with dual short-range/long-range\n   character that maps to multi-head attention's dual specialization.\n\n---\n\n## 2. Background and Related Work\n\n### 2.1 Graph Neural Networks\n\nGraph Neural Networks (GNNs) [Scarselli et al., 2009] generalize neural\nnetworks to graph-structured data. The message-passing paradigm [Gilmer et\nal., 2017] defines node updates as aggregations of neighbor representations.\n\nGraph Attention Networks (GATs) [Velickovic et al., 2018] introduce learned\nattention weights between connected nodes. Key limitations for our purposes:\n(a) attention is restricted to direct neighbors — no global context;\n(b) communities are not considered; (c) training labels required.\n\nGraphSAGE [Hamilton et al., 2017] samples and aggregates local neighborhoods\nbut similarly lacks global structural awareness.\n\nNone of these use community structure as an organizational principle for\nattention.\n\n### 2.2 Knowledge Graph Reasoning\n\nEarly KG reasoning systems used rule-based approaches (AMIE [Galarraga et al.,\n2013]) or embedding-based methods (TransE [Bordes et al., 2013], RotatE [Sun\net al., 2019]). These methods produce entity embeddings but do not perform\nmulti-hop traversal in the attention-mechanism sense.\n\nPath-based reasoning approaches (DeepPath [Xiong et al., 2017], MINERVA [Das\net al., 2018]) use reinforcement learning to find paths. These are closer to\nour goal but require training data and do not use community structure.\n\n### 2.3 LLM + KG Hybrid Systems\n\nKG-GPT [Yao et al., 2023], KGPT [Chen et al., 2020], and related systems\nconnect KGs to LLMs as retrieval stores. The LLM always performs the\nreasoning step; the KG is passive.\n\nGraphRAG [Edge et al., 2024] uses community detection (Leiden) to summarize\ngraph clusters as text, which is then passed to an LLM for RAG. This is the\nclosest existing work. However:\n- Communities are summarized as static text chunks, not used as attention heads\n- The LLM performs all reasoning\n- Paths are not returned or made interpretable\n- The system is not graph-agnostic (designed for Microsoft's pipeline)\n\nRAPTOR [Sarthi et al., 2024] builds hierarchical text clusters for RAG but\noperates on documents, not graphs, and uses tree structure rather than\ncommunity structure.\n\n### 2.4 Community Detection\n\nThe Louvain algorithm [Blondel et al., 2008] optimizes modularity greedily\nbut can produce internally disconnected communities. Leiden [Traag et al.,\n2019] fixes this with a refinement phase promoting internal connectivity.\nLabel Propagation [Raghavan et al., 2007] is fast and unsupervised but\nnon-deterministic and produces variable quality.\n\n**DSCF** (this work) combines LPA and Leiden signals simultaneously at each\nnode update, using a temperature-annealing schedule. See Section 4.2.\n\n**DSCF is non-deterministic** (inherits LPA's shuffle-order sensitivity).\nFor reproducible results: run 3-5 trials and select the partition with\nhighest modularity score. For production: seed the RNG and document the seed.\n\n---\n\n## 3. The Structural Equivalence\n\nWe establish a complete operational mapping between Transformer components\nand KG operations. The CEREBRUM Architecture\n\n### 4.1 Community-Structured Attention (CSA)\n\nCSA computes attention weights for graph traversal that incorporate both local graph topology and global community structure.\n\n**Attention weight formula:**\n\nFor entity $u$ attending to entity $v$ at traversal hop $k$:\n\n$$a(u, v, k) = \\sigma\\left( \\alpha \\cdot \\cos(\\vec{e}_u, \\vec{e}_v) + \\beta \\cdot S_{com}(u, v) + \\gamma \\cdot w_{rel} - \\delta \\cdot d_{norm}(u, v) + \\epsilon \\cdot \\phi(k) \\right)$$\n\nWhere:\n- $\\vec{e}$ is the entity embedding.\n- $S_{com}(u, v)$:\n  - $1.0$ if $\\text{community}(u) == \\text{community}(v)$\n  - $0.5$ if communities are adjacent\n  - $\\exp(-\\lambda \\cdot d_{com})$ otherwise\n- $w_{rel}$: weight per relation type.\n- $d_{norm}$: normalized shortest path length.\n- $\\phi(k)$: hop decay (e.g., $1 / (1 + k)$).\n- $\\sigma$: sigmoid activation.\n\n\n**Default parameter values (zero-shot deployment):**\n- α = 0.4 (embedding similarity)\n- β = 0.4 (community membership)\n- γ = 0.1 (edge type)\n- δ = 0.05 (distance penalty)\n- ε = 0.05 (hop decay)\n\nParameters can be learned from (query, answer) pairs for supervised settings.\n\n**community_score definition (complete):**\n\n```\ncommunity_score(u, v):\n  if community(u) == community(v):          return 1.0\n  if communities_are_adjacent(u, v):        return 0.5\n  else:\n    d = community_distance(u, v)\n    return exp(-λ · d)\n\ncommunity_distance(u, v):\n  # Shortest path (in hops) between the community of u and community of v\n  # in the community-level graph, where communities are nodes and two\n  # communities are adjacent if ≥1 cross-community edge exists between them.\n  # Precomputed once after DSCF converges via BFS on the community graph.\n  # λ = 0.5 default (controls cross-community decay rate)\n```\n\n**Why this is not a GAT:**\n\nGATs compute `a(u, v) = f(Wu · emb(u), Wv · emb(v))` — purely from learned\nweights on adjacent node pairs. They cannot express the community membership\nterm β · community_score(u, v), which introduces global structural awareness\nwithout requiring the full O(n²) attention of Transformers.\n\nCSA is O(n · k̄ · C) where k̄ is average degree and C is the average number\nof community-adjacent entities to consider — far cheaper than Transformer\nattention while capturing global structure via community membership.\n\n### 4.2 Dual-Signal Community Fusion (DSCF)\n\nDSCF is the community detection algorithm that produces the attention head\nstructure. It is a key contribution in its own right.\n\n**The core innovation:** at each individual node update, both the LPA\nmajority-vote signal (local topology) and the modularity gain signal (global\nstructure) are computed. The decision incorporates both simultaneously,\ngoverned by a temperature parameter:\n\n```\nFor each node v at each iteration:\n\n  1. LPA signal:\n     lpa_cid = argmax over neighbor labels (majority vote)\n     lpa_conf = vote_count[lpa_cid] / total_neighbors  ∈ [0, 1]\n\n  2. Modularity signal:\n     For each candidate community $C$ adjacent to $v$:\n     $$\\Delta Q(v \\to C) = \\frac{k_{v,C}}{m} - \\gamma \\frac{k_v \\sum k_C}{2m^2}$$\n     $\\text{best\\_mod\\_cid} = \\text{argmax } \\Delta Q$\n     $\\text{mod\\_conf} = \\min(\\text{best\\_}\\Delta Q \\cdot m, 1.0)$\n\n  3. Decision:\n     if lpa_cid == best_mod_cid ≠ current:\n       MOVE (consensus anchor — high confidence)\n     elif both say STAY:\n       STAY\n     elif only LPA says MOVE:\n       MOVE with probability lpa_conf × temperature\n     elif only modularity says MOVE:\n       MOVE with probability mod_conf × (1 + (1 − temperature))\n     else (disagree on different targets):\n       lpa_weight = lpa_conf × temperature\n       mod_weight = mod_conf × (2 − temperature)\n       MOVE to weighted-random choice\n\n  temperature schedule: τ_{t+1} = max(τ_t × cooling, 0.01)\n```\n\nPost-convergence: Leiden-style connectivity check — split any community whose\ninduced subgraph is disconnected.\n\n**Why DSCF communities are the right attention heads:**\n\nIn a trained Transformer:\n- Some heads specialize on local structure (syntactic patterns, adjacent tokens)\n- Other heads specialize on long-range structure (coreference, semantic themes)\n- The combination allows both local and global reasoning simultaneously\n\nDSCF communities exhibit exactly this dual character. The LPA component supports\ncommunities are locally coherent (nodes in the same community are topologically\nclose). The modularity component supports communities are globally significant\n(they represent structurally distinct regions of the graph). A node that is in\na DSCF community is there because *both* local and global signals agreed.\n\nThis dual property has not previously been used as a basis for attention\nmechanisms in any published graph learning system.\n\n**Comparison to Leiden-only communities:**\n\nLeiden optimizes purely for modularity. On sparse regions of a graph, Leiden\nmay split locally coherent neighborhoods across multiple communities because\nthe global modularity gain favors a different partition. DSCF resists this —\nthe LPA component holds locally coherent groups together even when modularity\nwould split them.\n\n**Comparison to LPA-only communities:**\n\nLPA can merge structurally distinct regions if they happen to be locally\nconnected (the \"resolution limit\" problem). DSCF resists this — the\nmodularity signal penalizes over-merging.\n\n### 4.3 Probabilistic Traversal: Bayesian Beam Search (v1.1.0)\n\nTo handle topological noise and cold-start uncertainty, CEREBRUM v1.1.0 introduces a probabilistic mode where edge weights are modeled as Beta distributions:\n\n$$P(a | \\alpha, \\beta) = \\frac{a^{\\alpha-1} (1-a)^{\\beta-1}}{B(\\alpha, \\beta)}$$\n\nDuring traversal, Thompson Sampling is employed to select neighbors. For each candidate neighbor $v$:\n1. A sample $x$ is drawn from $\\text{Beta}(\\alpha_v, \\beta_v)$.\n2. Neighbors are ranked by $x$, and the top-$B$ candidates are retained.\n\n**Warm-Starting**: To reduce initial variance, the distribution is seeded using the deterministic CSA score $s$:\n$$\\alpha_{init} = s \\cdot \\omega, \\quad \\beta_{init} = (1-s) \\cdot \\omega$$\nwhere $\\omega$ is the `warm_start_strength` (default 10.0).\n\n### 4.4 Cross-Modal Alignment: Procrustes SVD (v1.1.0)\n\n`SignalEncoder` aligns non-textual signals (e.g., FFT spectra of sensor data) into the canonical entity embedding space $\\mathcal{E}$ using Orthogonal Procrustes Analysis. Given a set of $n$ anchor points $X$ in the signal space and their corresponding entity embeddings $Y$ in $\\mathcal{E}$, we find the optimal rotation matrix $R$:\n\n$$R = \\arg\\min_{\\Omega^T\\Omega=I} \\| \\Omega X - Y \\|_F$$\n\nThe solution is obtained via SVD of the covariance matrix $M = Y X^T$:\n$$M = U \\Sigma V^T \\implies R = U V^T$$\n\n**Canonical Basis Anchor**: To prevent geometric drift in federated hops, all alignments are anchored to a fixed root embedding space $\\mathcal{E}_{root}$ rather than chaining through intermediate adapters.\n\n### 4.5 Federated Lease & Pinning (v1.1.0)\n\nIn high-velocity streaming environments, remote nodes may be evicted before a multi-hop federated query completes. `FederatedAdapter` implements a lease protocol:\n1. During the discovery phase, the local node sends a `PIN(entity_id, ttl)` request to the remote peer.\n2. The remote peer \"pins\" the entity, exempting it from sliding-window eviction for the duration of the TTL.\n3. The lease is automatically released upon TTL expiry or explicit `UNPIN` from the local node.\n\n### 4.6 Cryptographic Path Provenance: HMAC-SHA256\n\nTo prevent adversarial path injection in federated reasoning, CEREBRUM implements HMAC-based verification for remote adapter responses. A shared secret $\\mathcal{K}$ is configured between peers. \n\nFor a response body $\\mathcal{M}$ (the list of candidate edges), the remote peer computes the signature $\\mathcal{S}$:\n$$\\mathcal{S} = \\text{HMAC-SHA256}(\\mathcal{K}, \\mathcal{M})$$\n\nThe local adapter verifies the signature by computing $\\mathcal{S}' = \\text{HMAC-SHA256}(\\mathcal{K}, \\mathcal{M})$ and checking $\\mathcal{S}' \\equiv \\mathcal{S}$ using a constant-time comparison to prevent timing attacks. Responses with missing or invalid signatures are discarded.\n\n### 4.7 Lazy STDP Weight Decay\n\nThe $O(N)$ complexity of global weight decay in the STDP engine is addressed via a lazy update mechanism. For a causal pair $(u, v)$, the system stores the causal weight $w_{uv}$ and the global step counter $t_{last}$ at which the pair was last updated.\n\nGiven a current global step $T$ and decay rate $\\lambda \\in [0, 1]$, the current weight $w'_{uv}$ is computed only upon access:\n$$w'_{uv} = w_{uv} \\cdot \\lambda^{(T - t_{last})}$$\n\nThis transforms the global $O(N)$ maintenance task into a local $O(1)$ operation, enabling sub-millisecond event processing even on massive, highly-active graphs.\n\n---\n\n## 5. The Forward Pass: Graph Reasoning\n\n### 5.1 Algorithm\n\n```\nINPUT:  Query Q (text string or entity list)\n        Graph G (any backend)\n        Community assignments C (from DSCF)\n        Entity embeddings E (from any KGE method)\n        Hop depth L, beam width B, top-K\n\nOUTPUT: Ranked list of reasoning paths P = [(path, score, explanation)]\n\nSTEP 1 — Entity Grounding\n  If Q is text: extract entities via NER or fuzzy match to graph vocabulary\n  S = {e₁, e₂, ..., eₙ}  // seed entities\n  h⁰ᵢ = E[eᵢ]             // initial embeddings\n\nSTEP 2 — Structural Encoding\n  For each seed entity eᵢ:\n    pos_i = [pagerank(eᵢ), betweenness(eᵢ), degree(eᵢ), community_id(eᵢ)]\n    h⁰ᵢ = LayerNorm(h⁰ᵢ + W_pos · pos_i)\n\nSTEP 3 — Attention Traversal (L layers)\n  beam = [(path=[eᵢ], embedding=h⁰ᵢ, score=1.0) for each eᵢ in S]\n\n  For k = 1 to L:\n    candidates = []\n    For each (path, h, score) in beam:\n      current = path[-1]\n      neighbors = G.neighbors(current)\n\n      For each neighbor v in neighbors:\n        w = CSA(current, v, k)                    // attention weight\n        h_new = ReLU(W_k · (w · E[v] + h))       // aggregation\n        // W_k: hop-depth projection matrix (dim → dim)\n        // Zero-shot default: W_k = I (identity) — no learned projection\n        // Supervised setting: W_k learned per-hop via path-ranking loss\n        h_new += h                                 // residual\n        h_new = LayerNorm(h_new)\n        path_score = score × w × community_coherence(path + [v])\n        candidates.append((path + [v], h_new, path_score))\n\n    beam = top_B(candidates, key=path_score)      // beam pruning\n\nSTEP 4 — Path Scoring\n  Final score for path P = (e₁ → r₁ → e₂ → r₂ → ... → eₗ):\n    score(P) = Π attention_weights\n             × community_coherence(P)\n             × semantic_alignment(h_final, query_embedding)\n\nSTEP 5 — Output\n  Return top-K paths ranked by score\n  Each path includes:\n    - Ordered entity/relation sequence\n    - Score breakdown (attention, community, semantic)\n    - Community sequence (which \"heads\" were traversed)\n    - Natural language explanation template\n\n**Example Reasoning Trace:**\n\nQuery: *\"Who discovered a radioactive element?\"*\n\n1.  **Entity Grounding**: `Marie Curie` is identified as the seed node.\n2.  **Hop 1 (Attention)**: `Marie Curie` **—[discovered]→** `Polonium`.\n    - *CSA Weight*: High due to same community membership (Scientific Discoveries) and semantic alignment.\n3.  **Hop 2 (Traversal)**: `Polonium` **—[is_a]→** `Radioactive Element`.\n    - *CSA Weight*: High due to relation type weights and embedding proximity.\n4.  **Output**: The path `Marie Curie → discovered → Polonium → is_a → Radioactive Element` is returned as a grounded proof.\n```\n\n### 5.2 Community Coherence\n\nThe `community_coherence` term rewards paths that traverse communities in a\nprincipled way:\n\n```\ncommunity_coherence(path) =\n    (intra-community steps × 1.0 + cross-community steps × 0.5)\n    / total steps\n```\n\nA path that stays within one community scores 1.0 (tight local reasoning).\nA path that makes one community transition scores ~0.75 (one conceptual leap).\nThis prevents paths that jump incoherently across unrelated domains.\n\n### 5.3 Interpretability\n\nThe output of CEREBRUM is always a path through the KG. This is fundamentally\ndifferent from LLM reasoning in two ways:\n\n1. **Verifiable**: every step is an explicit (entity, relation, entity) triple\n   that exists in the graph. The system cannot hallucinate a connection that\n   isn't there.\n\n2. **Auditable**: the community sequence tells you *which conceptual domains*\n   were traversed. A path that crosses from community \"Clinical Trials\" to\n   community \"Drug Mechanisms\" to community \"Side Effects\" is immediately\n   understandable.\n\nThis property makes CEREBRUM specifically valuable in high-stakes domains\n(medical, legal, financial) where hallucination is unacceptable.\n\n---\n\n## 6. Complexity\n\n**DSCF community detection**: O(E · I) where E = edges, I = iterations\n(typically I < 50 before convergence). Comparable to Louvain.\n\n**Community graph construction**: O(E) post-DSCF; precomputed once.\n\n**CSA weight per edge**: O(d) where d = embedding dimension. Precompute\nstructural encodings once; community lookups are O(1) with a hash table.\n\n**Beam traversal (one query)**: O(B · L · k̄ · d) where:\n- B = beam width (default 10)\n- L = hop depth (default 3)\n- k̄ = average degree\n- d = embedding dimension\n\nFor a graph with k̄=20, L=3, B=10, d=384: ~230,000 floating-point operations\nper query — milliseconds on CPU, no GPU required.\n\n**Comparison to Transformer**: Full attention is O(n² · d) per layer.\nCEREBRUM traversal is O(B · L · k̄ · d) — independent of graph size n.\nThis makes CEREBRUM sublinear in graph size for fixed-width beam search.\n\n### 6.7 Known Failure Modes\n\n**Dense hub nodes**: Nodes with very high degree (k >> k̄) cause beam\nexplosion at that hop. Mitigation: cap `max_neighbors` per node (default 50).\n\n**Homogeneous graphs**: If all nodes share the same community, CSA degenerates\nto pure embedding similarity (community_score terms cancel). This occurs in\nhighly regular graphs (grids, complete bipartite). Mitigation: check community\ncount post-DSCF; if count < 3, fall back to BFS with embedding similarity only.\n\n**Disconnected graphs**: Multiple connected components each get their own\ncommunities; cross-component traversal is impossible by definition. Queries\nspanning components will return no paths. Mitigation: surface component\nboundaries to callers; allow separate per-component queries.\n\n**Sparse embedding coverage**: Entities with generic or missing labels get\nnear-random sentence embeddings. The α term (embedding similarity) becomes\nnoise; DSCF community structure (β term) carries the full weight. Still\nfunctions; interpretability of similarity scores is reduced.\n\n**Adversarial community injection**: A malicious actor who can insert edges\ninto the graph can manipulate community structure and thus attention weights.\nRelevant for applications where the KG is user-writable. Mitigation: sign\ntrusted edges; treat unsigned-edge-induced communities with lower β weight.\n\n### 6.8 Horizontal Scalability\n\n**BLUF: CEREBRUM query serving scales horizontally without architectural limit. Queries are stateless reads from a sharded graph database; throughput grows linearly with workers. LLM inference cannot do this.**\n\nCEREBRUM's scalability is bounded by the scalability of graph databases — a solved problem at web scale — not by the constraints of neural network inference.\n\n#### Three independent parallelism dimensions\n\n**Within a single query.** At each beam hop, all B×k̄ candidate edge scores are computed independently. The only synchronization point is the top-B selection (O(B log B) on ~200 items). Every other operation — embedding lookup, cosine similarity, CSA weight computation — is embarrassingly parallel across GPU threads.\n\n```\nHop k:\n  Score(candidate_1 × neighbors)  ──┐\n  Score(candidate_2 × neighbors)  ──┤→ top-B select → next hop\n  ...                               │\n  Score(candidate_B × neighbors)  ──┘\n  (only sync point: O(B log B) reduce)\n```\n\n**Across concurrent queries.** Traversal workers are stateless during inference — the graph and embedding matrix are read-only. Multiple concurrent queries share no mutable state. Query throughput scales linearly with worker count up to I/O bandwidth limits, with zero inter-query synchronization overhead.\n\n**Across machines.** The CSA formula strongly prefers staying within a community (community_score = 1.0 intra vs. exponential decay inter). This makes communities the natural shard key: most queries never cross a shard boundary and therefore never leave their home machine. Cross-shard traversal is a sparse, predictable network call — already implemented in the FederatedAdapter. The holographic index (Bloom filter + community centroid) lets any machine identify which remote shard owns an entity without loading remote data.\n\n#### Amdahl's Law comparison\n\n| | LLM inference | CEREBRUM query |\n|---|---|---|\n| Sequential fraction | ~100% of forward pass (layer N waits for layer N-1) | Top-B select at each of L hops (~3μs total) |\n| Parallel fraction | ~0% within a single inference call | ~99.9% of all scoring work |\n| Cross-machine sync | All-reduce at every layer boundary | Sparse shard crossings (most queries: zero) |\n| Practical speedup ceiling | ~8–16 GPUs per inference call (communication dominates) | Linear with worker count up to DB I/O limits |\n\n#### Scaling tiers\n\n| Scale | Graph storage | Compute | Status |\n|---|---|---|---|\n| Millions of nodes | NetworkXAdapter (RAM) | Single machine, GPU embeddings | Implemented |\n| Billions of edges | Neo4jAdapter (disk) | Single machine | Adapter exists |\n| Multi-machine partitioned | FederatedAdapter (community-sharded) | Cluster of API instances | Phase 6–9 |\n| 100B+ edges | Distributed graph DB (Neptune, TigerGraph) + stateless Kubernetes workers | Cluster | Architecture compatible; adapter needed |\n| Web scale (1T+ edges) | Distributed graph processing (Spark GraphX) for offline DSCF; standard graph DB for queries | Datacenter | Theoretical; DSCF distribution is the hard problem |\n\nThe bottleneck at extreme scale shifts entirely to community detection — DSCF requires a global view of the graph to compute modularity gain. Distributed community detection (distributed Louvain, FENNEL-style partitioning) is an active research area. Critically, community detection runs **offline and periodically** — queries continue running against the last known partition while the next detection pass runs in the background. It is never on the query critical path.\n\n**The one-line summary: a KG query touches ~0.001% of the graph per call using embarrassingly parallel operations. An LLM touches 100% of its parameters every time, sequentially. That gap defines the scalability story.**\n\n---\n\n## 7. Open Research Questions\n\n### 9.1 Embedding Strategy\n\nTwo options exist:\n\n**Option A — Pre-trained structural embeddings (TransE/RotatE)**: trained\non the graph structure itself. More precise but requires a training step.\nSuitable for static KGs or when training compute is available.\n\n**Option B — On-the-fly label embeddings (sentence-transformers)**: encode\nentity labels and descriptions using a pre-trained language model. No graph-\nspecific training needed. More agnostic. Less precise for entities with\nambiguous labels.\n\n**Recommended default**: Option B for zero-shot deployment; Option A when the\ngraph has been stable and training is feasible. CEREBRUM should support both\ninterchangeably via the EmbeddingEngine interface.\n\n### 8.2 Adaptive Community Granularity\n\nThe DSCF resolution parameter controls how many communities are formed. Too\nfew communities = coarse attention heads that miss structure. Too many = noisy\nheads that don't generalize.\n\n**Proposed adaptive rule**: target K ≈ √N communities, where N = node count.\nThis is consistent with theoretical results on optimal modularity resolution\nand supports attention head count scales sensibly with graph size.\n\nFor Home Assistant's KG (N ≈ 5,000): target ~70 communities.\nFor Wikidata subset (N ≈ 100,000): target ~316 communities.\n\n### 8.3 Soft vs Hard Community Membership\n\nDSCF produces hard assignments (each node belongs to exactly one community).\nReal-world entities often span multiple communities — a person can be both\na scientist and a politician.\n\n**Extension**: weight-based soft membership, where each node has a probability\ndistribution over communities. The community_score function becomes a\ndot product of membership vectors. This would require modifying DSCF to track\nconfidence scores at convergence.\n\n### 8.4 Learnable Parameters\n\nIn zero-shot mode, α, β, γ, δ, ε are fixed. For supervised settings, they\ncan be learned from (query, ground-truth-answer) pairs via gradient descent\non a path-ranking loss. This is an optional enhancement that does not affect\nthe core architecture.\n\n### 8.5 Temporal Knowledge Graphs\n\nTime-stamped KGs (events, evolving relationships) introduce a temporal\ndimension. The positional encoding would need to incorporate temporal distance\nalongside graph-structural distance. Left for future work.\n\n---\n\n## Acknowledgments: Intellectual Debt and Credits\n\nCEREBRUM stands on the shoulders of decades of research in graph theory, community detection, and neural networks. This evolution is designed to close the **\"Mesoscale Gap\"**—the structural region between immediate local topology (LPA) and global modularity (Leiden).\n\n### The Motivation for a Third Signal\n\nWhile modularity (Global) captures static edge density and LPA (Local) captures immediate neighborhood cohesion, they both miss the **dynamic flow of information** through a network. In reasoning tasks, the most relevant path is often the one that information naturally \"flows\" along.\n\n### The TSC Components\n\n1.  **LPA (Local)**: Neighbor recognition (Cohesion).\n2.  **Modularity (Global)**: Architecture optimization (Significance).\n3.  **Infomap / Map Equation (Mid-Level)**: Flow-based clustering (Connectivity). Originally proposed by Martin Rosvall and Carl Bergstrom (2008), Infomap uses random walks to identify sub-clusters based on information flow, acting as the \"mesoscale\" judge between local and global signals.\n\n### Consensus Decision Logic\n\nIn the TSC framework, a node move or a traversal edge must pass a **Consensus Filter**. This reduces \"structural hallucinations\"—paths that exist topologically but lack conceptual or informational flow coherence.\n\nThe fused probability for a node move or attention weight calculation becomes:\n$$P(\\text{move}) = f(\\text{LPA} \\cdot \\tau_{local}, \\text{Mod} \\cdot \\tau_{global}, \\text{Infomap} \\cdot \\tau_{mid})$$\n\nThis \"mid-level voting\" helps confirm that only the most structurally and dynamically robust reasoning chains survive the beam-search pruning process. TSC will be implemented as an optional, high-precision mode within the CEREBRUM core, allowing for direct comparison with DSCF.\n\n### TSC Development Roadmap (Phases 6–10)\n\n| Phase | Title | Primary Research Objective |\n| :--- | :--- | :--- |\n| **Phase 6** | **Flow Integration** | Formalize $P(\\text{move}) = f(\\text{LPA, Mod, Infomap})$ decision logic. |\n| **Phase 7** | **TSC Engine** | Implement Infomap signal processing and Consensus Confidence scoring. |\n| **Phase 8** | **Dynamic Traversal**| Implement consensus-gated Beam Search and `flow_coherence` ranking. |\n| **Phase 9** | **Ablation H4** | Prove precision superiority of TSC over DSCF on complex topologies. |\n| **Phase 10**| **v0.2.0 Release** | Productionize high-precision reasoning for high-stakes deployment. |\n\n---\n\n---\n\n## 10. Every step in a legal argument must be citable; CEREBRUM's\ngrounded paths match this requirement exactly.\n\n**Cybersecurity**: Attack graphs, CVE dependency networks. \"What path leads\nfrom this exposed service to root access?\" — a life-safety question that\nbenefits from verified, traceable reasoning chains.\n\n**Software engineering**: Code dependency and call graphs. Impact analysis:\n\"What does changing function X affect?\" traversed as a multi-hop attention\npath with community context (same module = high attention).\n\n**Finance**: Entity relationship graphs for regulatory compliance. Traceable\nreasoning chains for auditors: \"Why did this transaction trigger a flag?\"\n\n### 10.2 The LLM Bridge\n\nCEREBRUM is designed to augment LLMs, not replace them. The `llm_bridge`\nmodule formats traversal output as structured context:\n\n```\nYou are reasoning about: [query]\n\nThe knowledge graph traversal found these paths:\n\nPath 1 (score: 0.94):\n  Marie Curie [COMMUNITY: Scientific Discoveries]\n  → [discovered] →\n  Polonium [COMMUNITY: Scientific Discoveries]\n  → [exhibits] →\n  Radioactivity [COMMUNITY: Physics Phenomena]\n\nPlease summarize what this tells us about [query] in natural language.\n```\n\nThis gives any LLM a grounded, structured context that minimizes the risk of\nhallucination because the facts are provided explicitly. The LLM's role is\npurely natural language generation, not reasoning.\n\n### 10.3 The Agnosticism Property\n\nCEREBRUM is agnostic across five dimensions:\n\n1. **Graph database**: implement GraphAdapter for any system\n2. **Embedding method**: implement EmbeddingEngine for any model\n3. **LLM**: any model or none — CEREBRUM works without one\n4. **Domain**: the algorithm is domain-blind; community structure emerges\n   from the graph's own topology\n5. **Query language**: entities can be identified from text, IDs, or direct\n   lookup — the entry point is flexible\n\n---\n\n## 11. Production Hardening (Phase 10)\n\nPhase 10 transitioned CEREBRUM from research prototype to deployable service.\n\n### 11.1 JWT Authentication\n\nAll REST endpoints require a Bearer JWT. Tokens are validated by a FastAPI\ndependency; expired tokens return HTTP 403, missing tokens return HTTP 401.\nToken issuance is external, enabling OAuth2/OIDC integration.\n\n### 11.2 ResourceGovernor\n\nPer-request resource ceilings prevent query runaway:\n\n$$\\text{ResourceGovernor: } \\{N_{\\max}, E_{\\max}, B_{\\max}, H_{\\max}, T_{\\max}, C_{\\max}\\}$$\n\nwhere $N_{\\max}$ = max nodes, $E_{\\max}$ = max edges, $B_{\\max}$ = beam width,\n$H_{\\max}$ = max hops, $T_{\\max}$ = timeout (seconds), $C_{\\max}$ = concurrent\nqueries. Violations return HTTP 429 with a structured JSON error body.\n\n### 11.3 Asynchronous Beam Traversal\n\n`AsyncBeamTraversal` wraps `BeamTraversal` in `asyncio` and adds a\n`/query/stream` SSE endpoint. One event is emitted per beam step:\n\n```\nevent: step\ndata: {\"hop\": k, \"candidates\": [...], \"scores\": [...]}\n\nevent: result\ndata: {\"paths\": [...], \"top_score\": 0.94, \"hops\": 2}\n```\n\nThe `ResourceGovernor` timeout applies per stream connection.\n\n---\n\n## 12. Real-Time Streaming (Phase 11)\n\nPhase 11 extends CEREBRUM to continuous, live data. The streaming subsystem\nsits above the core CSA/DSCF algorithms, which are unchanged.\n\n### 12.1 Architecture Layers\n\n| Layer | Component | Responsibility |\n|---|---|---|\n| Ingestion | `StreamSource` subclasses | FileTail, HTTP polling, WebSocket, MQTT, Python callbacks |\n| Discretization | `*Discretizer` classes | Continuous signal → discrete graph edges |\n| Buffering | `SlidingWindowBuffer` | Time-window + count cap + reference counting |\n| Live graph | `StreamAdapter` | Thread-safe `NetworkXAdapter` with mutation broadcast |\n| Community | `IncrementalCommunityUpdater` | Ego-network DSCF on affected nodes only |\n| API | `/stream/*` endpoints | Ingest, status, SSE event subscription |\n\n### 12.2 Sliding Window Buffer\n\nThe buffer maintains live edges satisfying two simultaneous constraints:\n\n$$\\mathcal{W}(t) = \\left\\{ e \\in \\mathcal{E} \\;\\middle|\\; t - t_e \\leq \\Delta t \\right\\} \\cap \\left\\{ |\\mathcal{E}| \\leq N_{\\max} \\right\\}$$\n\nReference counting allows the same edge $(s, r, t)$ to be held by multiple\noverlapping events; the edge is evicted only when all references expire.\n\n### 12.3 Signal Discretization\n\n**ThresholdDiscretizer** — categorical states with hysteresis:\n\n$$\\text{state}(x) = \\begin{cases} \\text{SPIKE} & x > \\mu + 3\\sigma \\\\ \\text{HIGH} & x > \\mu + \\sigma \\\\ \\text{LOW} & x < \\mu - \\sigma \\\\ \\text{NORMAL} & \\text{otherwise} \\end{cases}$$\n\n**BinningDiscretizer** — uniform quantization into $N$ bins:\n\n$$\\text{bin}(x) = \\left\\lfloor \\frac{x - x_{\\min}}{x_{\\max} - x_{\\min}} \\cdot N \\right\\rfloor$$\n\n**CoActivationDiscretizer** — emit `CO_ACTIVATES` when two sensors fire within\nwindow $\\delta_t$ and have co-activated at least $n_{\\min}$ times:\n\n$$\\text{emit}(A,B) \\iff |t_A - t_B| \\leq \\delta_t \\;\\wedge\\; \\text{count}(A,B) \\geq n_{\\min}$$\n\n**TemporalSequenceDiscretizer** — `PRECEDES` chains between consecutive events.\n\n**ObjectDetectionDiscretizer** — `DETECTS` + `CO_OCCURS_WITH` from frame detections.\n\n### 12.4 Incremental Community Updates\n\nFull DSCF after each ingestion event is prohibitive. The updater restricts\nre-computation to the ego-network of radius $r$ around affected nodes:\n\n$$G_{\\text{local}} = \\{v \\mid d_G(v, u) \\leq r,\\; u \\in \\mathcal{A}\\}$$\n\nUpdates trigger when $|\\mathcal{A}| \\geq n_{\\min}$ (default 10). All\nassignments outside $G_{\\text{local}}$ are preserved from the previous full\nrun, maintaining a globally-consistent, locally-fresh partition.\n\n### 12.5 Streaming API\n\n| Endpoint | Method | Description |\n|---|---|---|\n| `/stream/ingest` | POST | Batch-ingest `StreamEvent` objects |\n| `/stream/status` | GET | Live stats (nodes, edges, events/s, communities) |\n| `/stream/events` | GET (SSE) | Subscribe to graph mutation events |\n\n---\n\n## 12.6 Bridge Twin Nodes (Phase 12)\n\nWhen a node crosses community boundaries repeatedly — because the graph's true reasoning path requires it — the CSA distance penalty imposes a persistent cost on an edge that is structurally correct. **Bridge Twins** resolve this by materialising a copy of the node inside the destination community once the crossing count and semantic fit threshold are both met:\n\n$$\\text{fit}(v, \\mathcal{C}_d) = \\cos\\!\\left(\\mathbf{e}_v,\\; \\frac{1}{|\\mathcal{C}_d|}\\sum_{u \\in \\mathcal{C}_d} \\mathbf{e}_u\\right) \\geq \\theta_{bridge}$$\n\nThe twin $v'$ is assigned $c(v') = \\mathcal{C}_d$, carries $\\mathbf{e}_{v'} = \\mathbf{e}_v$, and is connected via bidirectional `BRIDGE_TWIN` edges ($w_{rel} = 1.0$). The CSA short-circuits to $\\sigma(0.925 + \\varepsilon\\phi(k)) \\approx 0.716$ for bridge edges. Idle bridges are pruned after $\\tau_{prune}$ days (LTD analog). Biological correspondence: thalamic relay nuclei (LGN).\n\n---\n\n## 12.7 Spike-Timing Dependent Plasticity (Phase 13)\n\nThe `CoActivationDiscretizer` detects symmetric correlations. `STDPDiscretizer` adds **directional causal inference** from timing alone, matching the biological STDP rule:\n\n$$\\Delta w(A \\to B) = \\begin{cases}\nA_+ \\exp(-\\Delta t/\\tau_+) & \\Delta t > 0 \\quad \\text{(LTP: A precedes B)} \\\\\n-A_- \\exp(\\Delta t/\\tau_-) & \\Delta t \\leq 0 \\quad \\text{(LTD: anti-causal)}\n\\end{cases}$$\n\nDefault: $A_+ = 0.1$, $A_- = 0.105$ (slight LTD dominance, matching Bi & Poo 1998). A `CAUSES(A→B)` edge is emitted when $w(A \\to B) \\geq w_{threshold}$ AND $n(A \\to B) \\geq n_{min}$. Per-spike multiplicative weight decay ($\\lambda \\in (0,1]$) provides exponential forgetting. The result: CEREBRUM autonomously infers directed causal chains from streaming data without domain configuration or labeled training.\n\n---\n\n## 12.8 REM Cycle Memory Consolidation (Phase 14)\n\nThe `REMEngine` (`core/rem_engine.py`) implements a biologically-inspired offline memory consolidation pass modeled on NREM slow-wave sleep. It operates in three sequential phases: **Prune**, **Consolidate**, and **Synthesize**.\n\n**Prune** removes edges whose confidence has decayed below a floor threshold:\n\n$$\\text{prune}(e) \\iff \\text{confidence}(e) < \\theta_{prune} = 0.2 \\;\\wedge\\; \\text{type}(e) \\neq \\text{BRIDGE\\_TWIN}$$\n\n`BRIDGE_TWIN` edges are immune — they represent experience-confirmed structural relays (Phase 12) and must survive the prune pass.\n\n**Consolidate** re-runs DSCF on the pruned graph so that community boundaries re-align with the surviving, higher-confidence edge set.\n\n**Synthesize** scans all surviving entity pairs and emits a `rem_synthesized` edge when two conditions are jointly met:\n\n$$\\cos(\\mathbf{e}_u, \\mathbf{e}_v) \\geq \\theta_{sim} = 0.8 \\;\\wedge\\; d_G(u, v) \\leq h_{prox} = 4$$\n\nSynthesized edges carry `confidence = 0.3`, placing them just above the prune floor and making them candidates for validation by subsequent traversal.\n\n**Operational modes:**\n\n| Mode | Effect |\n|---|---|\n| `dry_run=True` | Zero mutations; returns full report of what would be pruned, re-consolidated, and synthesized |\n| `dry_run=False` | Mutations applied in-place |\n| `rollback()` | Restores pruned edges with original attributes; removes synthetic edges. One level of undo. |\n\n**API endpoints:**\n\n| Endpoint | Method | Description |\n|---|---|---|\n| `/rem/run` | POST | Execute REM cycle (`dry_run` param, default `true`) |\n| `/rem/rollback` | POST | Undo last non-dry-run cycle |\n| `/rem/status` | GET | Current REM state, last cycle stats |\n\nBiological analog: **NREM slow-wave sleep** — prune weak synapses (synaptic homeostasis), consolidate the community architecture, and form associative links between semantically proximate but topologically distant nodes.\n\n---\n\n## 12.9 InsightEngine — Surprise-Driven Discovery (Phase 15)\n\nThe `InsightEngine` (`core/insight_engine.py`) implements a continuous surprise-detection system over the reasoning stream. It fires an `InsightEvent` when a traversal path scores significantly above the rolling baseline for its community pair.\n\n**Surprise signal:**\n\n$$\\text{surprise}(P) = \\text{score}(P) - \\mu_{\\text{baseline}}(C_u, C_v)$$\n\nwhere $\\mu_{\\text{baseline}}$ is a per-community-pair rolling mean maintained in a fixed-size ring buffer (O(1) update). An `InsightEvent` fires when:\n\n$$\\text{surprise}(P) > \\theta_{salience} = 0.35$$\n\n**Insight score:**\n\n$$\\text{insight\\_score} = \\min\\!\\left(1,\\; \\frac{\\text{surprise} + \\text{explanatory\\_power}}{2}\\right)$$\n\n$$\\text{explanatory\\_power}(C_u, C_v) = \\frac{|C_u| \\times |C_v|}{N(N-1)/2}$$\n\nThe explanatory power term rewards insights that bridge large, structurally disconnected communities — high-value discoveries that span conceptual gaps.\n\n**Three-tier architecture:**\n\n| Tier | Mechanism | Cost |\n|---|---|---|\n| Hot path | O(1) ring-buffer surprise check per traversal step | ~0% CPU |\n| Warm path | Daemon thread, event queue drain + baseline update | ~0.01% CPU |\n| Cold path | Full community-boundary scan (scheduled) | ~50 ms/hr |\n\n**Materialization:** When an `InsightEvent` fires, the engine materializes an `INSIGHT_LINK` edge on the insight path:\n\n$$\\text{INSIGHT\\_LINK}: \\text{confidence} = 0.85,\\; w = 2.0,\\; \\text{provenance} = \\text{\"insight\"}$$\n\nBecause `confidence = 0.85 \\gg \\theta_{prune} = 0.2`, insight links **resist REM pruning** by a wide margin.\n\n**Hebbian reward propagation:** Every edge on the insight path receives a confidence boost:\n\n$$\\Delta\\text{confidence}(e) = \\delta_{hebbian} \\times \\text{insight\\_score}, \\quad \\text{capped at } 1.0$$\n\n**REM + Insight feedback loop:**\n\n$$\\underbrace{\\text{REM proposes}}_{\\text{confidence} = 0.3} \\xrightarrow{\\text{traversal validates}} \\underbrace{\\text{Insight cements}}_{\\text{confidence} = 0.85} \\xrightarrow{\\text{resists}} \\underbrace{\\text{next REM prune}}_{\\theta_{prune} = 0.2}$$\n\nThis loop implements a structural analog of the dopamine prediction-error signal + AMPA receptor upregulation cycle in biological memory consolidation.\n\n**API endpoints:**\n\n| Endpoint | Method | Description |\n|---|---|---|\n| `/insight/status` | GET | Engine state, event counts, baseline stats |\n| `/insight/events` | GET | Recent `InsightEvent` list |\n| `/insight/scan` | POST | Trigger immediate cold-path community-boundary scan |\n\nBiological analog: **dopamine prediction-error signal** (surprise fires when actual reward exceeds expected) combined with **AMPA receptor upregulation** (Hebbian boost to the edges that carried the surprising path).\n\n---\n\n## 12.10 Insight Validation and Metacognition (Phase 16)\n\nPhase 16 adds two components that operate above the InsightEngine: a structural verifier for individual discoveries and a second-order pattern detector across the discovery history.\n\n### 12.10.1 InsightValidator — Bilateral Verification and Triangulation\n\nA raw `InsightEvent` represents a *surprising* connection. Surprise is a necessary but insufficient condition for a discovery to be trusted. `InsightValidator` (`core/insight_validator.py`) applies two independent structural tests before promoting an insight's confidence.\n\n**Check 1 — Bilateral reverse traversal:**\n\nFor an insight connecting source $s$ to target $t$, the bilateral check asks whether the *original graph structure* (excluding the INSIGHT_LINK edge itself and the direct $s \\leftrightarrow t$ edge) can independently reach $s$ from $t$ within max_hop steps:\n\n$$\\text{bilateral}(s, t) = \\exists\\; \\text{path}_{G \\setminus \\{\\text{INSIGHT\\_LINK}, (s,t)\\}}(t \\to s,\\; \\text{length} \\leq h_{\\max})$$\n\nThe exclusion of both the INSIGHT_LINK edge and the direct pair prevents circular reasoning: the discovered connection cannot serve as its own evidence. If neither the insight link nor the original cross-edge can be used, a return path found in the remaining structure constitutes genuine independent confirmation.\n\n**Check 2 — Multi-path corroboration (triangulation):**\n\nUp to $N_{seeds}$ (default 5) other nodes in the same source community as $s$ are tested as independent starting points. The source node $s$ itself is removed from the graph to prevent corroborating paths from simply routing through $s$'s existing connection:\n\n$$\\text{corroboration}(t) = \\left|\\left\\{ u \\in \\mathcal{C}_s \\setminus \\{s, t\\} \\;\\middle|\\; \\exists\\; \\text{path}_{G_{\\text{und}} \\setminus \\{s\\}}(u \\to t,\\; \\text{length} \\leq h_{\\max}) \\right\\}\\right|$$\n\nIf multiple community-peers independently reach $t$, the structural robustness of the connection is confirmed — it is not an artifact of one unusual path from $s$.\n\n**Validation outcomes:**\n\n| Status | Condition | Confidence after validation |\n|---|---|---|\n| `corroborated` | bilateral ∧ corroboration ≥ 2 | 0.95 |\n| `bilateral` | bilateral ∧ corroboration < 2 | 0.92 |\n| `unilateral` | ¬bilateral ∧ corroboration ≥ 1 | unchanged (0.85) |\n| `isolated` | ¬bilateral ∧ corroboration = 0 | unchanged (0.85), flagged |\n\nConfidence promotion updates the INSIGHT_LINK edge in the graph so future traversal preferentially uses validated paths.\n\n**API endpoints:**\n\n| Endpoint | Method | Description |\n|---|---|---|\n| `/insight/validate/all` | POST | Validate all pending InsightEvents |\n| `/insight/validate/{event_id}` | POST | Validate a single InsightEvent by ID |\n\n### 12.10.2 MetaInsightEngine — Second-Order Reasoning\n\nThe `MetaInsightEngine` (`core/meta_insight_engine.py`) watches the stream of `InsightEvent` objects and maintains an **InsightGraph**: a directed NetworkX graph where nodes are InsightEvent IDs and edges are structural relationships between insights.\n\n**Four connection types** are detected for every pair of insights $(A, B)$:\n\n| Type | Condition | Score |\n|---|---|---|\n| `chain` | $A.\\text{target} = B.\\text{source}$ or $B.\\text{target} = A.\\text{source}$ | $\\frac{s_A + s_B}{2}$ |\n| `shared_entity` | $\\{s_A, t_A, \\text{bridge}_A\\} \\cap \\{s_B, t_B, \\text{bridge}_B\\} \\neq \\emptyset$ | score × 0.8 |\n| `community_overlap` | $\\text{leap}_A = \\text{leap}_B \\geq 1$ | score × 0.6 |\n| `temporal_cluster` | $|T_A - T_B| \\leq \\Delta t_{\\text{window}}$ | score × 0.4 |\n\nWhen an InsightGraph edge score exceeds `chain_score_threshold` (default 0.3), a `MetaInsightEvent` fires — the system has recognized a pattern in its own discovery history.\n\n**Depth-2 higher-order detection:**\n\nWhen a new insight $C$ arrives and finds predecessors $B$ who themselves have predecessors $A$, a depth-2 `MetaInsightEvent` fires:\n\n$$A \\xrightarrow{\\text{meta}} B \\xrightarrow{\\text{meta}} C \\implies \\text{MetaInsightEvent}(A, C,\\; \\text{depth}=2,\\; \\text{chain}=[A, B, C])$$\n\nThis corresponds to the recursive self-awareness expressed as \"I notice that my discoveries about $B$ and $C$ are themselves connected to an earlier discovery about $A$.\" At depth 2, the system is reasoning about relationships between relationships — the structural equivalent of analogical reasoning.\n\nThe biological correspondence is the formation of **episodic meta-memories**: the brain doesn't only remember individual experiences, it remembers that two experiences were related, and can later notice that this remembered relationship is itself related to a third. The InsightGraph is the structural substrate of this capacity in CEREBRUM.\n\n**API endpoints:**\n\n| Endpoint | Method | Description |\n|---|---|---|\n| `/meta-insight/status` | GET | Engine state, total meta-events, InsightGraph size |\n| `/meta-insight/events` | GET | Recent `MetaInsightEvent` list |\n| `/meta-insight/graph` | GET | Full InsightGraph as JSON (nodes + edges) |\n\n**Biological analog:** Formation of **episodic memory clusters** in the hippocampus — the binding of related memories into schemata, and subsequent schema-to-schema associations during memory consolidation. Phase 16 closes the loop started in Phase 14 (REM offline consolidation) and Phase 15 (online surprise detection): discoveries are now verified before being trusted, and the pattern of verified discoveries is itself analyzed for higher-order structure.\n\n---\n\n## 13. kv = degree[v]\n\n      // LPA signal\n      vote = COUNT(assignment[nb] for nb in neighbors)\n      lpa_cid = argmax(vote); lpa_conf = vote[lpa_cid] / |neighbors|\n\n      // Modularity signal\n      candidates = {assignment[nb] for nb in neighbors} - {cur}\n      best_cid = cur; best_dq = 0\n      FOR cid IN candidates:\n        k_vc = |{nb in neighbors : assignment[nb] == cid}|\n        dq = k_vc/m - resolution × kv × com_k[cid] / (2m²)\n        IF dq > best_dq: best_dq = dq; best_cid = cid\n      mod_conf = min(best_dq × m, 1.0)\n\n      // Decision\n      IF lpa_cid == best_cid != cur:\n        new = lpa_cid  // consensus anchor\n      ELIF best_cid == cur AND lpa_cid == cur:\n        CONTINUE       // both say stay\n      ELIF best_cid == cur:\n        IF random() >= lpa_conf × temperature: CONTINUE\n        new = lpa_cid\n      ELIF lpa_cid == cur:\n        IF random() >= mod_conf × (1 + (1−temperature)): CONTINUE\n        new = best_cid\n      ELSE:\n        lpa_w = lpa_conf × temperature\n        mod_w = mod_conf × (2 − temperature)\n        new = weighted_choice({lpa_cid: lpa_w, best_cid: mod_w})\n\n      com_k[cur] = max(com_k[cur] − kv, 0)\n      com_k[new] = com_k[new] + kv\n      assignment[v] = new; changed = True\n\n    temperature = max(temperature × cooling, 0.01)\n    IF NOT changed: BREAK\n\n  // Connectivity post-pass (Leiden-style)\n  // First component keeps original ID; additional components get new IDs.\n  // (Preserves ID stability for majority partition; important for\n  //  community_score lookup caching.)\n  RETURN [component for community in assignment.values()\n          for component in connected_components(G.subgraph(community))]\n```\n\n---\n\n## Appendix B: CSA Weight Formula — Parameter Sensitivity\n\nThe default parameter values (α=0.4, β=0.4, γ=0.1, δ=0.05, ε=0.05) were\nchosen based on the following intuitions:\n\n- Embedding similarity (α) and community membership (β) are given equal weight\n  because both capture complementary aspects of relevance: similarity captures\n  semantic proximity while community captures structural proximity.\n- Edge type (γ) is given lower weight because it is most useful in domain-\n  specific settings with rich edge type vocabularies.\n- Distance penalty (δ) is kept small to allow multi-hop paths without excessive\n  pruning.\n- Hop decay (ε) is minimal to allow deep traversal when needed.\n\nIn practice, α + β dominate the attention weights for most graphs.\n\n---\n\n## Appendix C: Relationship to Existing KG Embedding Methods\n\nTransE [Bordes et al., 2013] represents relations as translations in embedding\nspace: emb(h) + emb(r) ≈ emb(t) for (h, r, t) triples. CEREBRUM can use\nTransE embeddings directly for the similarity term in CSA without modification.\n\nRotatE [Sun et al., 2019] represents relations as rotations in complex space.\nMore expressive for symmetric, antisymmetric, and compositional relations.\nAlso directly usable in CEREBRUM.\n\nNeither TransE nor RotatE produces multi-hop reasoning paths on their own.\nCEREBRUM uses their embeddings as the semantic grounding layer while the\ntraversal logic and community structure provide the reasoning.\n\n---\n\n## Appendix D: Prototype Code (Deprecated)\n\n> **Note**: The prototype code previously contained in this appendix has been\n> superseded by the production implementation in `core/community_engine.py`.\n> Refer to **Appendix A** for the current TSC algorithm pseudocode.\n\n---\n\n## Appendix E: Project Bootstrap Guide\n\n### E.1 Phase 0 Completion Status\n\nPhase 0 is **complete**.
