These methods produce entity embeddings but do not perform\nmulti-hop traversal in the attention-mechanism sense.\n\nPath-based reasoning approaches (DeepPath [Xiong et al., 2017], MINERVA [Das\net al., 2018]) use reinforcement learning to find paths. These are closer to\nour goal but require training data and do not use community structure.\n\n### 2.3 LLM + KG Hybrid Systems\n\nKG-GPT [Yao et al., 2023], KGPT [Chen et al., 2020], and related systems\nconnect KGs to LLMs as retrieval stores. The CEREBRUM Architecture\n\n### 4.1 Community-Structured Attention (CSA)\n\nCSA computes attention weights for graph traversal that incorporate both local\ngraph topology and global community structure.\n\n**Attention weight formula:**\n\nFor entity u attending to entity v at traversal hop k:\n\n```\na(u, v, k) = σ(\n    α · cosine_sim(emb(u), emb(v))\n  + β · community_score(u, v)\n  + γ · edge_type_weight(type(u → v))\n  - δ · normalized_distance(u, v)\n  + ε · hop_decay(k)\n)\n```\n\nWhere:\n- `emb(·)` is the entity embedding (any KGE method or sentence encoder)\n- `community_score(u, v)`:\n  - 1.0 if community(u) == community(v)           [same head]\n  - 0.5 if communities are adjacent               [neighboring heads]\n  - exp(-λ · community_distance(u, v)) otherwise  [distance decay]\n- `edge_type_weight(type)`: learned or manually assigned per relation type\n- `normalized_distance(u, v)`: shortest path length / graph diameter\n- `hop_decay(k)`: encourages shorter paths (e.g., 1 / (1 + k))\n- σ: sigmoid activation\n- α, β, γ, δ, ε: tunable parameters\n\n**Default parameter values (zero-shot deployment):**\n- α = 0.4 (embedding similarity)\n- β = 0.4 (community membership)\n- γ = 0.1 (edge type)\n- δ = 0.05 (distance penalty)\n- ε = 0.05 (hop decay)\n\nParameters can be learned from (query, answer) pairs for supervised settings.\n\n**community_score definition (complete):**\n\n```\ncommunity_score(u, v):\n  if community(u) == community(v):          return 1.0\n  if communities_are_adjacent(u, v):        return 0.5\n  else:\n    d = community_distance(u, v)\n    return exp(-λ · d)\n\ncommunity_distance(u, v):\n  # Shortest path (in hops) between the community of u and community of v\n  # in the community-level graph, where communities are nodes and two\n  # communities are adjacent if ≥1 cross-community edge exists between them.\n  # Precomputed once after DSCF converges via BFS on the community graph.\n  # λ = 0.5 default (controls cross-community decay rate)\n```\n\n**Why this is not a GAT:**\n\nGATs compute `a(u, v) = f(Wu · emb(u), Wv · emb(v))` — purely from learned\nweights on adjacent node pairs. They cannot express the community membership\nterm β · community_score(u, v), which introduces global structural awareness\nwithout requiring the full O(n²) attention of Transformers.\n\nCSA is O(n · k̄ · C) where k̄ is average degree and C is the average number\nof community-adjacent entities to consider — far cheaper than Transformer\nattention while capturing global structure via community membership.\n\n### 4.2 Dual-Signal Community Fusion (DSCF)\n\nDSCF is the community detection algorithm that produces the attention head\nstructure. It is a key contribution in its own right.\n\n**The core innovation:** at each individual node update, both the LPA\nmajority-vote signal (local topology) and the modularity gain signal (global\nstructure) are computed. The decision incorporates both simultaneously,\ngoverned by a temperature parameter:\n\n```\nFor each node v at each iteration:\n\n  1. LPA signal:\n     lpa_cid = argmax over neighbor labels (majority vote)\n     lpa_conf = vote_count[lpa_cid] / total_neighbors  ∈ [0, 1]\n\n  2. Modularity signal:\n     For each candidate community C adjacent to v:\n       ΔQ(v→C) = k_{v,C}/m − resolution × k_v × Σk_C / (2m²)\n     best_mod_cid = argmax ΔQ\n     mod_conf = min(best_ΔQ × m, 1.0)  ∈ [0, 1]\n\n  3. Decision:\n     if lpa_cid == best_mod_cid ≠ current:\n       MOVE (consensus anchor — high confidence)\n     elif both say STAY:\n       STAY\n     elif only LPA says MOVE:\n       MOVE with probability lpa_conf × temperature\n     elif only modularity says MOVE:\n       MOVE with probability mod_conf × (1 + (1 − temperature))\n     else (disagree on different targets):\n       lpa_weight = lpa_conf × temperature\n       mod_weight = mod_conf × (2 − temperature)\n       MOVE to weighted-random choice\n\n  temperature schedule: τ_{t+1} = max(τ_t × cooling, 0.01)\n```\n\nPost-convergence: Leiden-style connectivity check — split any community whose\ninduced subgraph is disconnected.\n\n**Why DSCF communities are the right attention heads:**\n\nIn a trained Transformer:\n- Some heads specialize on local structure (syntactic patterns, adjacent tokens)\n- Other heads specialize on long-range structure (coreference, semantic themes)\n- The combination allows both local and global reasoning simultaneously\n\nDSCF communities exhibit exactly this dual character. The LPA component supports\ncommunities are locally coherent (nodes in the same community are topologically\nclose). The modularity component supports communities are globally significant\n(they represent structurally distinct regions of the graph). A node that is in\na DSCF community is there because *both* local and global signals agreed.\n\nThis dual property has not previously been used as a basis for attention\nmechanisms in any published graph learning system.\n\n**Comparison to Leiden-only communities:**\n\nLeiden optimizes purely for modularity. On sparse regions of a graph, Leiden\nmay split locally coherent neighborhoods across multiple communities because\nthe global modularity gain favors a different partition. DSCF resists this —\nthe LPA component holds locally coherent groups together even when modularity\nwould split them.\n\n**Comparison to LPA-only communities:**\n\nLPA can merge structurally distinct regions if they happen to be locally\nconnected (the \"resolution limit\" problem). DSCF resists this — the\nmodularity signal penalizes over-merging.\n\n---\n\n## 5. The Forward Pass: Graph Reasoning\n\n### 5.1 Algorithm\n\n```\nINPUT:  Query Q (text string or entity list)\n        Graph G (any backend)\n        Community assignments C (from DSCF)\n        Entity embeddings E (from any KGE method)\n        Hop depth L, beam width B, top-K\n\nOUTPUT: Ranked list of reasoning paths P = [(path, score, explanation)]\n\nSTEP 1 — Entity Grounding\n  If Q is text: extract entities via NER or fuzzy match to graph vocabulary\n  S = {e₁, e₂, ..., eₙ}  // seed entities\n  h⁰ᵢ = E[eᵢ]             // initial embeddings\n\nSTEP 2 — Structural Encoding\n  For each seed entity eᵢ:\n    pos_i = [pagerank(eᵢ), betweenness(eᵢ), degree(eᵢ), community_id(eᵢ)]\n    h⁰ᵢ = LayerNorm(h⁰ᵢ + W_pos · pos_i)\n\nSTEP 3 — Attention Traversal (L layers)\n  beam = [(path=[eᵢ], embedding=h⁰ᵢ, score=1.0) for each eᵢ in S]\n\n  For k = 1 to L:\n    candidates = []\n    For each (path, h, score) in beam:\n      current = path[-1]\n      neighbors = G.neighbors(current)\n\n      For each neighbor v in neighbors:\n        w = CSA(current, v, k)                    // attention weight\n        h_new = ReLU(W_k · (w · E[v] + h))       // aggregation\n        // W_k: hop-depth projection matrix (dim → dim)\n        // Zero-shot default: W_k = I (identity) — no learned projection\n        // Supervised setting: W_k learned per-hop via path-ranking loss\n        h_new += h                                 // residual\n        h_new = LayerNorm(h_new)\n        path_score = score × w × community_coherence(path + [v])\n        candidates.append((path + [v], h_new, path_score))\n\n    beam = top_B(candidates, key=path_score)      // beam pruning\n\nSTEP 4 — Path Scoring\n  Final score for path P = (e₁ → r₁ → e₂ → r₂ → ... → eₗ):\n    score(P) = Π attention_weights\n             × community_coherence(P)\n             × semantic_alignment(h_final, query_embedding)\n\nSTEP 5 — Output\n  Return top-K paths ranked by score\n  Each path includes:\n    - Ordered entity/relation sequence\n    - Score breakdown (attention, community, semantic)\n    - Community sequence (which \"heads\" were traversed)\n    - Natural language explanation template\n```\n\n### 5.2 Community Coherence\n\nThe `community_coherence` term rewards paths that traverse communities in a\nprincipled way:\n\n```\ncommunity_coherence(path) =\n    (intra-community steps × 1.0 + cross-community steps × 0.5)\n    / total steps\n```\n\nA path that stays within one community scores 1.0 (tight local reasoning).\nA path that makes one community transition scores ~0.75 (one conceptual leap).\nThis prevents paths that jump incoherently across unrelated domains.\n\n### 5.3 Interpretability\n\nThe output of CEREBRUM is always a path through the KG. This is fundamentally\ndifferent from LLM reasoning in two ways:\n\n1. **Verifiable**: every step is an explicit (entity, relation, entity) triple\n   that exists in the graph. The system cannot hallucinate a connection that\n   isn't there.\n\n2. **Auditable**: the community sequence tells you *which conceptual domains*\n   were traversed. A path that crosses from community \"Clinical Trials\" to\n   community \"Drug Mechanisms\" to community \"Side Effects\" is immediately\n   understandable.\n\nThis property makes CEREBRUM specifically valuable in high-stakes domains\n(medical, legal, financial) where hallucination is unacceptable.\n\n---\n\n## 6. 6.6 Computational Complexity\n\n**DSCF community detection**: O(E · I) where E = edges, I = iterations\n(typically I < 50 before convergence). Comparable to Louvain.\n\n**Community graph construction**: O(E) post-DSCF; precomputed once.\n\n**CSA weight per edge**: O(d) where d = embedding dimension. Precompute\nstructural encodings once; community lookups are O(1) with a hash table.\n\n**Beam traversal (one query)**: O(B · L · k̄ · d) where:\n- B = beam width (default 10)\n- L = hop depth (default 3)\n- k̄ = average degree\n- d = embedding dimension\n\nFor a graph with k̄=20, L=3, B=10, d=384: ~230,000 floating-point operations\nper query — milliseconds on CPU, no GPU required.\n\n**Comparison to Transformer**: Full attention is O(n² · d) per layer.\nCEREBRUM traversal is O(B · L · k̄ · d) — independent of graph size n.\nThis makes CEREBRUM sublinear in graph size for fixed-width beam search.\n\n### 6.7 Known Failure Modes\n\n**Dense hub nodes**: Nodes with very high degree (k >> k̄) cause beam\nexplosion at that hop. Mitigation: cap `max_neighbors` per node (default 50).\n\n**Homogeneous graphs**: If all nodes share the same community, CSA degenerates\nto pure embedding similarity (community_score terms cancel). This occurs in\nhighly regular graphs (grids, complete bipartite). Mitigation: check community\ncount post-DSCF; if count < 3, fall back to BFS with embedding similarity only.\n\n**Disconnected graphs**: Multiple connected components each get their own\ncommunities; cross-component traversal is impossible by definition. Queries\nspanning components will return no paths. Mitigation: surface component\nboundaries to callers; allow separate per-component queries.\n\n**Sparse embedding coverage**: Entities with generic or missing labels get\nnear-random sentence embeddings. The α term (embedding similarity) becomes\nnoise; DSCF community structure (β term) carries the full weight. Still\nfunctions; interpretability of similarity scores is reduced.\n\n**Adversarial community injection**: A malicious actor who can insert edges\ninto the graph can manipulate community structure and thus attention weights.\nRelevant for applications where the KG is user-writable. Mitigation: sign\ntrusted edges; treat unsigned-edge-induced communities with lower β weight.\n\n---\n\n## 7. The DSCF-as-Attention-Head Hypothesis\n\nThe central theoretical claim of CEREBRUM is that DSCF communities are better\nattention heads than Leiden-only or LPA-only communities. We state this as a\nfalsifiable hypothesis:\n\n**H1 (DSCF Attention Hypothesis)**: For multi-hop reasoning tasks on KGs,\nCEREBRUM with DSCF attention heads achieves higher answer accuracy than\nCEREBRUM with Leiden-only or LPA-only attention heads.\n\n**H2 (CSA vs GAT Hypothesis)**: CSA-guided traversal achieves higher accuracy\non multi-hop questions than GAT-based traversal on the same graph and same\nentity embeddings.\n\n**H3 (Interpretability Hypothesis)**: CEREBRUM paths receive higher human\ncoherence ratings than equivalent LLM-generated reasoning chains on the same\nquestions, because every step is a grounded graph edge.\n\n**H3 Evaluation Protocol**: Present matched pairs — one CEREBRUM path and\none LLM reasoning chain — to N≥30 annotators blind to their source. Ask:\n\"Which reasoning chain is more coherent and trustworthy? (A / B / Equal)\".\nPrimary metric: proportion preferring CEREBRUM path. Secondary: Cohen's kappa\nfor inter-annotator agreement (target κ > 0.6).\n\nThese hypotheses are testable on standard benchmarks (WebQSP, MetaQA-3hop)\nand define the empirical work for Phase 2.\n\n---\n\n## 8. Open Research Questions\n\n### 8.1 Embedding Strategy\n\nTwo options exist:\n\n**Option A — Pre-trained structural embeddings (TransE/RotatE)**: trained\non the graph structure itself. More precise but requires a training step.\nSuitable for static KGs or when training compute is available.\n\n**Option B — On-the-fly label embeddings (sentence-transformers)**: encode\nentity labels and descriptions using a pre-trained language model. No graph-\nspecific training needed. More agnostic. Less precise for entities with\nambiguous labels.\n\n**Recommended default**: Option B for zero-shot deployment; Option A when the\ngraph has been stable and training is feasible. CEREBRUM should support both\ninterchangeably via the EmbeddingEngine interface.\n\n### 8.2 Adaptive Community Granularity\n\nThe DSCF resolution parameter controls how many communities are formed. Too\nfew communities = coarse attention heads that miss structure. Too many = noisy\nheads that don't generalize.\n\n**Proposed adaptive rule**: target K ≈ √N communities, where N = node count.\nThis is consistent with theoretical results on optimal modularity resolution\nand supports attention head count scales sensibly with graph size.\n\nFor Home Assistant's KG (N ≈ 5,000): target ~70 communities.\nFor Wikidata subset (N ≈ 100,000): target ~316 communities.\n\n### 8.3 Soft vs Hard Community Membership\n\nDSCF produces hard assignments (each node belongs to exactly one community).\nReal-world entities often span multiple communities — a person can be both\na scientist and a politician.\n\n**Extension**: weight-based soft membership, where each node has a probability\ndistribution over communities. The community_score function becomes a\ndot product of membership vectors. This would require modifying DSCF to track\nconfidence scores at convergence.\n\n### 8.4 Learnable Parameters\n\nIn zero-shot mode, α, β, γ, δ, ε are fixed. For supervised settings, they\ncan be learned from (query, ground-truth-answer) pairs via gradient descent\non a path-ranking loss. This is an optional enhancement that does not affect\nthe core architecture.\n\n### 8.5 Temporal Knowledge Graphs\n\nTime-stamped KGs (events, evolving relationships) introduce a temporal\ndimension. The positional encoding would need to incorporate temporal distance\nalongside graph-structural distance. Left for future work.\n\n---\n\n## 9. Broader Impact and Applications\n\n### 10.1 Domain Applications\n\n**Biomedical**: Drug-gene-disease-pathway graphs. Multi-hop reasoning for drug\nrepurposing (\"Drug X inhibits enzyme Y which is overexpressed in disease Z\").\nGrounded inference is critical — no LLM should hallucinate drug interactions.\n\n**Legal**: Case law citation and statutory reference networks. Multi-hop\nprecedent tracing. Every step in a legal argument must be citable; CEREBRUM's\ngrounded paths match this requirement exactly.\n\n**Cybersecurity**: Attack graphs, CVE dependency networks. \"What path leads\nfrom this exposed service to root access?\" — a life-safety question that\nbenefits from verified, traceable reasoning chains.\n\n**Software engineering**: Code dependency and call graphs. Impact analysis:\n\"What does changing function X affect?\" traversed as a multi-hop attention\npath with community context (same module = high attention).\n\n**Finance**: Entity relationship graphs for regulatory compliance. Traceable\nreasoning chains for auditors: \"Why did this transaction trigger a flag?\"\n\n### 10.2 The LLM Bridge\n\nCEREBRUM is designed to augment LLMs, not replace them. The `llm_bridge`\nmodule formats traversal output as structured context:\n\n```\nYou are reasoning about: [query]\n\nThe knowledge graph traversal found these paths:\n\nPath 1 (score: 0.94):\n  Marie Curie [COMMUNITY: Scientific Discoveries]\n  → [discovered] →\n  Polonium [COMMUNITY: Scientific Discoveries]\n  → [exhibits] →\n  Radioactivity [COMMUNITY: Physics Phenomena]\n\nPlease summarize what this tells us about [query] in natural language.\n```\n\nThis gives any LLM a grounded, structured context that minimizes the risk of\nhallucination because the facts are provided explicitly. The LLM's role is\npurely natural language generation, not reasoning.\n\n### 10.3 The Agnosticism Property\n\nCEREBRUM is agnostic across five dimensions:\n\n1. **Graph database**: implement GraphAdapter for any system\n2. **Embedding method**: implement EmbeddingEngine for any model\n3. **LLM**: any model or none — CEREBRUM works without one\n4. **Domain**: the algorithm is domain-blind; community structure emerges from the graph's own topology\n5. **Query language**: entities can be identified from text, IDs, or direct lookup — the entry point is flexible\n\n### 10.4 Engram: AI-to-AI Knowledge Sync (Phase 45)\n\nAs Knowledge Graphs scale, passing full natural language reasoning paths to LLMs becomes token-prohibitive. In Phase 45, we introduced the **Engram shorthand** dialect — a high-density symbolic shorthand designed for machine-to-machine reasoning transfer.\n\nEngram shorthand achieves **30x compression** by mapping relation types to single-character tokens (e.g., `!` for CAUSES, `~` for INFLUENCED) and truncating entity labels. This allows an LLM to consume a complete multi-hop reasoning trace in under 50 tokens, compared to ~500 for natural language.\n\n**Engram Symbolic Mapping Examples:**\n- `!` : CAUSES\n- `+` : TREATS\n- `*` : STARRED_IN\n- `~` : INFLUENCED\n- `≈` : REM_SYNTHESIZED (Structural Similarity)\n\nExample Engram trace: `Engram:[Lansop!>Steven!>Trimet!>Agranu(c0.81)]`\n(Translation: Lansoprazole causes Stevens-Johnson syndrome which causes Trimethoprim which causes Agranulocytosis).\n\n---\n\n## 11. Conclusion\n\n\nWe have presented CEREBRUM: a framework that enables Knowledge Graphs to\nreason using the structural principles of Transformer attention without\ntraining data, without an LLM, and with full interpretability.\n\nThe two core contributions — Community-Structured Attention (CSA) and\nDual-Signal Community Fusion (DSCF) — work together to give a KG the dual\ncharacter of multi-head attention: local cohesion (from DSCF's LPA component)\ncombined with global structural significance (from DSCF's modularity\ncomponent).\n\nThe resulting system produces reasoning paths, not **Black-Box** embeddings. Every\nanswer is traceable to a sequence of verified graph edges. This architectural shift \nmoves AI from probabilistic hidden-layer weights to a **Glass-Box** of deterministic \npaths — a vital transition in the modern AI/ML landscape. Every reasoning step\nnames the community it traversed. This interpretability property, combined with\nthe graph-grounded capability of graph-grounded inference, positions CEREBRUM\nas a meaningful complement to — and in certain domains, replacement for —\nLLM-based reasoning over structured knowledge.\n\nThe open questions identified in Section 8 define the research program. The\nbenchmarks in Section 9 define the empirical standard. The architecture in\nSection 6 defines what to build.\n\nThe name CEREBRUM refers to the optical phenomenon where two viewpoints on\nthe same object yield depth perception that neither viewpoint alone provides.\nLPA and modularity are two viewpoints on the same graph. Their combination\nyields structural depth — attention heads with both short-range and long-range\ncharacter — that neither produces alone. This multi-signal consensus is inspired \nby **mid-level voting** systems in triplex-redundant aircraft navigation, where \nthe median value is selected to correct navigation errors. CEREBRUM applies this \nprinciple to \"right the navigation errors\" (hallucinations) of current language \nmodels by requiring structural consensus for every reasoning step.\n\nThat depth is what makes the KG reason.\n\n---\n\n## Acknowledgments: Intellectual Debt and Credits\n\nCEREBRUM stands on the shoulders of decades of research in graph theory, community detection, and neural networks. Single-pass neighbor analysis (O(k))\n      counts = {}; cent_sums = {}\n      FOR nb IN neighbors:\n        cid = assignment[nb]\n        counts[cid]++\n        IF centrality_weights: cent_sums[cid] += centrality_weights[nb]\n\n      // 2. LPA signal\n      lpa_cid = argmax(counts); lpa_conf = counts[lpa_cid] / |neighbors|\n\n      // 3. Modularity signal\n      best_mod_cid = cur; best_dq = 0\n      FOR cid, k_vc IN counts:\n        IF cid == cur: CONTINUE\n        dq = k_vc/m - resolution × kv × com_k[cid] / (2m²)\n        IF dq > best_dq: best_dq = dq; best_mod_cid = cid\n      mod_conf = min(best_dq × m, 1.0)\n\n      // 4. Centrality Signal (TSC exclusive)\n      IF centrality_weights:\n        cent_cid = argmax(cent_sums)\n        cent_conf = cent_sums[cent_cid] / sum(cent_sums)\n      ELSE:\n        cent_cid, cent_conf = lpa_cid, lpa_conf\n\n      // 5. Consensus Logic\n      // Check if ANY signal wants to move\n      IF lpa_cid == cur AND best_mod_cid == cur AND cent_cid == cur:\n        CONTINUE\n\n      // Check for Full Consensus Move (Anchor)\n      IF lpa_cid == best_mod_cid == cent_cid:\n        new = lpa_cid\n      ELSE:\n        // Weighted probabilistic choice\n        lpa_w = lpa_conf × temperature\n        mod_w = mod_conf × (2 − temperature)\n        cent_w = cent_conf × (1.5 - temperature)\n\n        // Boost pairwise consensus\n        IF lpa_cid == cent_cid: lpa_w *= 1.5; cent_w *= 1.5\n        IF lpa_cid == best_mod_cid: lpa_w *= 1.5; mod_w *= 1.5\n        // ... (etc)\n\n        new = weighted_choice({lpa_cid: lpa_w, best_mod_cid: mod_w, cent_cid: cent_w})\n\n      IF new != cur:\n        com_k[cur] -= kv; com_k[new] += kv\n        assignment[v] = new; changed = True\n\n    temperature = max(temperature × cooling, 0.01)\n    IF NOT changed: BREAK\n\n  RETURN leiden_post_pass(G, assignment)\n```\n      neighbors = N(v)\n      IF |neighbors| == 0: CONTINUE\n\n      cur = assignment[v]; kv = degree[v]\n\n      // LPA signal\n      vote = COUNT(assignment[nb] for nb in neighbors)\n      lpa_cid = argmax(vote); lpa_conf = vote[lpa_cid] / |neighbors|\n\n      // Modularity signal\n      candidates = {assignment[nb] for nb in neighbors} - {cur}\n      best_cid = cur; best_dq = 0\n      FOR cid IN candidates:\n        k_vc = |{nb in neighbors : assignment[nb] == cid}|\n        dq = k_vc/m - resolution × kv × com_k[cid] / (2m²)\n        IF dq > best_dq: best_dq = dq; best_cid = cid\n      mod_conf = min(best_dq × m, 1.0)\n\n      // Decision\n      IF lpa_cid == best_cid != cur:\n        new = lpa_cid  // consensus anchor\n      ELIF best_cid == cur AND lpa_cid == cur:\n        CONTINUE       // both say stay\n      ELIF best_cid == cur:\n        IF random() >= lpa_conf × temperature: CONTINUE\n        new = lpa_cid\n      ELIF lpa_cid == cur:\n        IF random() >= mod_conf × (1 + (1−temperature)): CONTINUE\n        new = best_cid\n      ELSE:\n        lpa_w = lpa_conf × temperature\n        mod_w = mod_conf × (2 − temperature)\n        new = weighted_choice({lpa_cid: lpa_w, best_cid: mod_w})\n\n      com_k[cur] = max(com_k[cur] − kv, 0)\n      com_k[new] = com_k[new] + kv\n      assignment[v] = new; changed = True\n\n    temperature = max(temperature × cooling, 0.01)\n    IF NOT changed: BREAK\n\n  // Connectivity post-pass (Leiden-style)\n  // First component keeps original ID; additional components get new IDs.\n  // (Preserves ID stability for majority partition; important for\n  //  community_score lookup caching.)\n  RETURN [component for community in assignment.values()\n          for component in connected_components(G.subgraph(community))]\n```\n\n---\n\n## Appendix B: CSA Weight Formula — Parameter Sensitivity\n\nThe default parameter values (α=0.4, β=0.4, γ=0.1, δ=0.05, ε=0.05) were\nchosen based on the following intuitions:\n\n- Embedding similarity (α) and community membership (β) are given equal weight\n  because both capture complementary aspects of relevance: similarity captures\n  semantic proximity while community captures structural proximity.\n- Edge type (γ) is given lower weight because it is most useful in domain-\n  specific settings with rich edge type vocabularies.\n- Distance penalty (δ) is kept small to allow multi-hop paths without excessive\n  pruning.\n- Hop decay (ε) is minimal to allow deep traversal when needed.\n\nIn practice, α + β dominate the attention weights for most graphs.\n\n---\n\n## Appendix C: Relationship to Existing KG Embedding Methods\n\nTransE [Bordes et al., 2013] represents relations as translations in embedding\nspace: emb(h) + emb(r) ≈ emb(t) for (h, r, t) triples. CEREBRUM can use\nTransE embeddings directly for the similarity term in CSA without modification.\n\nRotatE [Sun et al., 2019] represents relations as rotations in complex space.\nMore expressive for symmetric, antisymmetric, and compositional relations.\nAlso directly usable in CEREBRUM.\n\nNeither TransE nor RotatE produces multi-hop reasoning paths on their own.\nCEREBRUM uses their embeddings as the semantic grounding layer while the\ntraversal logic and community structure provide the reasoning.\n\n---\n\n## Appendix D: Prototype Code (Deprecated)\n\n> **Note**: The prototype code previously contained in this appendix has been\n> superseded by the production implementation in `core/community_engine.py`.\n> Refer to **Appendix A** for the current TSC algorithm pseudocode.\n\n---\n\n## Appendix E: Project Bootstrap Guide\n\n### E.1 Phase 0 Completion Status\n\nPhase 0 is **complete**.
